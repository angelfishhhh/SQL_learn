## 1. 题目信息

---

## 2. 题目类型

一级类型：聚合查询 / 自连接  
二级类型：GROUP BY 分组 / AVG 平均值 / ROUND 保留小数  

本题类型：根据同一台机器、同一个进程的 `start` 和 `end` 两条记录，计算每个进程耗时，再按机器求平均耗时。

---

## 3. 表结构与粒度分析

表名：Activity

字段：

- machine_id：机器 ID
- process_id：进程 ID
- activity_type：活动类型，只可能是 `'start'` 或 `'end'`
- timestamp：当前时间，单位为秒

表粒度：

Activity 表一行代表某台机器的某个进程的一次活动记录。

一个完整进程会有两行：

```text
一行 start
一行 end
```

是否涉及 join：

是，可以使用自连接。

join 后行数是否可能变多：

会。

如果只按：

```sql
a1.machine_id = a2.machine_id
AND a1.process_id = a2.process_id
```

连接，那么同一个进程的两行记录会互相匹配，形成 4 行：

```text
start - start
start - end
end   - start
end   - end
```

所以必须限制：

```sql
a1.activity_type = 'start'
AND a2.activity_type = 'end'
```

这样每个进程只保留一行有效记录。

---

## 4. 目标输出分析

最终结果一行代表什么：

一行代表一台机器的平均进程处理时间。

需要输出哪些字段：

- machine_id
- processing_time

是否需要去重：

不需要。

是否需要分组：

需要。

按 `machine_id` 分组。

是否需要排序 / 排名：

不需要。题目要求结果可以按任意顺序返回。

---

## 5. 我的错误 SQL

```sql
SELECT a1.machine_id, ROUND(SUM(ABS(a1.timestamp - a2.timestamp)) / 4, 3) AS processing_time
FROM Activity AS a1
JOIN Activity AS a2
ON a1.machine_id = a2.machine_id
    AND a1.process_id = a2.process_id
GROUP BY machine_id
;
```

错误点 1：自连接后没有区分 `start` 和 `end`

你只按照 `machine_id` 和 `process_id` 连接，会导致每个进程产生 4 行：

```text
start - start
start - end
end   - start
end   - end
```

其中只有 `start - end` 才是我们真正需要的。

错误点 2：`ABS(a1.timestamp - a2.timestamp)` 是在补救错误连接

因为没有限制谁是开始、谁是结束，所以用了 `ABS()` 取绝对值。

但本题的正确逻辑应该是：

```sql
end.timestamp - start.timestamp
```

而不是让所有组合都参与计算。

错误点 3：`/ 4` 是硬编码，不可靠

在样例中，每个进程错误连接后产生 4 行，所以你用 `/ 4` 可能看起来合理。

但正确逻辑不是除以 4，而是：

```sql
AVG(每个进程的结束时间 - 开始时间)
```

也就是每个进程先算一次耗时，再对机器求平均。

---

## 6. 正确 SQL

```sql
SELECT 
    a1.machine_id,
    ROUND(AVG(a2.timestamp - a1.timestamp), 3) AS processing_time
FROM Activity AS a1
JOIN Activity AS a2
ON a1.machine_id = a2.machine_id
    AND a1.process_id = a2.process_id
    AND a1.activity_type = 'start'
    AND a2.activity_type = 'end'
GROUP BY a1.machine_id;
```

---

## 7. 解题思路

先把一条 `start` 记录和它对应的 `end` 记录连接到同一行。

连接条件是：

```sql
a1.machine_id = a2.machine_id
AND a1.process_id = a2.process_id
```

但这还不够，还要指定：

```sql
a1.activity_type = 'start'
AND a2.activity_type = 'end'
```

这样一行就代表一个完整进程。

然后用：

```sql
a2.timestamp - a1.timestamp
```

计算这个进程的处理时间。

最后按机器分组：

```sql
GROUP BY a1.machine_id
```

再求平均值：

```sql
AVG(a2.timestamp - a1.timestamp)
```

并保留 3 位小数：

```sql
ROUND(..., 3)
```

---

## 8. 本题易错点总结

这题的关键不是先 `SUM`，而是先识别一对 `start` 和 `end`。

看到一张表里同一个实体有两种状态：

```text
start
end
```

通常要想到两种做法：

第一种是自连接，把 `start` 和 `end` 拼成一行。

第二种是条件聚合，用 `CASE WHEN` 或 `IF` 把 `start` 当成负数，`end` 当成正数。

本题建议优先掌握自连接写法，因为它更直观：

```text
start 行 + end 行 → 一个完整进程 → 求耗时 → 按机器求平均
```

---

## 9. 另一种写法：条件聚合

```sql
SELECT
    machine_id,
    ROUND(
        AVG(
            CASE 
                WHEN activity_type = 'start' THEN -timestamp
                ELSE timestamp
            END
        ) * 2,
        3
    ) AS processing_time
FROM Activity
GROUP BY machine_id;
```

这个写法的含义是：

对于每台机器，每个进程有两行：

```text
start 记为 -timestamp
end   记为 +timestamp
```

所以同一个进程两行相加后就是：

```text
end timestamp - start timestamp
```

因为 `AVG()` 是对所有 `start` 和 `end` 行求平均，每个进程有 2 行，所以最后需要乘以 2。

---

## 10. 记忆模板

遇到同一张表中有开始和结束两种记录时，可以优先考虑自连接：

```sql
SELECT
    start_table.分组字段,
    ROUND(AVG(end_table.时间字段 - start_table.时间字段), 3) AS 结果字段
FROM 表名 AS start_table
JOIN 表名 AS end_table
ON start_table.匹配字段1 = end_table.匹配字段1
    AND start_table.匹配字段2 = end_table.匹配字段2
    AND start_table.状态字段 = 'start'
    AND end_table.状态字段 = 'end'
GROUP BY start_table.分组字段;
```

## 5. 我的错误 SQL

```sql
SELECT e.student_id, s.student_name, e.subject_name, COUNT(e.subject_name) AS attended_exams
FROM Examinations AS e
JOIN Students AS s
ON s.student_id = e.student_id
ORDER BY student_id
;
```

错误点 1：从 `Examinations` 表出发

你的 SQL 是从 `Examinations` 表开始查：

```sql
FROM Examinations AS e
```

但 `Examinations` 表只记录了已经参加过考试的数据。

如果某个学生没有参加某门考试，那么这张表里根本没有对应记录。

而题目要求：

```text
即使参加次数为 0，也要显示出来
```

所以不能只从 `Examinations` 表出发。

---

错误点 2：没有使用 `Subjects` 表

题目要求统计每个学生参加每门科目的次数。

这里的“每门科目”来自 `Subjects` 表。

你的 SQL 只用了：

```text
Examinations
Students
```

没有使用：

```text
Subjects
```

因此无法生成所有学生和所有科目的完整组合。

---

错误点 3：使用了 `JOIN`，会过滤掉没有考试记录的情况

你的 SQL 使用的是：

```sql
JOIN Students AS s
ON s.student_id = e.student_id
```

这里的 `JOIN` 等价于 `INNER JOIN`。

`INNER JOIN` 只会保留匹配成功的记录。

如果某个学生没有参加某门考试，那么这组记录不会出现在 `Examinations` 表里，也就不会出现在最终结果中。

但本题要求参加次数为 0 的情况也要显示，所以后面连接考试记录时应该使用：

```sql
LEFT JOIN
```

---

错误点 4：缺少 `GROUP BY`

你使用了聚合函数：

```sql
COUNT(e.subject_name)
```

同时又查询了普通字段：

```sql
e.student_id,
s.student_name,
e.subject_name
```

这种情况下应该使用 `GROUP BY`。

否则 SQL 逻辑不完整。

正确分组应该是：

```sql
GROUP BY
    s.student_id,
    s.student_name,
    sub.subject_name
```

---

错误点 5：排序条件不完整

题目要求按照：

```sql
student_id,
subject_name
```

排序。

你的 SQL 只写了：

```sql
ORDER BY student_id
```

缺少：

```sql
subject_name
```

## 6. 正确 SQL

```sql
SELECT
    s.student_id,
    s.student_name,
    sub.subject_name,
    COUNT(e.subject_name) AS attended_exams
FROM Students AS s
CROSS JOIN Subjects AS sub
LEFT JOIN Examinations AS e
ON s.student_id = e.student_id
    AND sub.subject_name = e.subject_name
GROUP BY
    s.student_id,
    s.student_name,
    sub.subject_name
ORDER BY
    s.student_id,
    sub.subject_name;
```

---

## 7. 解题思路

本题的关键是：

不能只统计 `Examinations` 表。

因为 `Examinations` 表只记录已经参加过考试的记录。

如果某个学生没有参加某门考试，那么 `Examinations` 表中没有对应行。

但题目要求：

```text
参加次数为 0 的情况也要显示
```

所以第一步应该先生成完整的结果框架：

```text
所有学生 × 所有科目
```

使用：

```sql
FROM Students AS s
CROSS JOIN Subjects AS sub
```

这样可以得到每个学生和每个科目的组合。

然后再用 `LEFT JOIN` 去连接实际考试记录：

```sql
LEFT JOIN Examinations AS e
ON s.student_id = e.student_id
    AND sub.subject_name = e.subject_name
```

这里必须同时匹配两个条件：

```text
学生相同
科目相同
```

最后用：

```sql
COUNT(e.subject_name)
```

统计考试次数。

如果没有匹配到考试记录，`e.subject_name` 是 `NULL`。

而：

```sql
COUNT(字段名)
```

不会统计 `NULL`，所以结果会是 0。

---

## 8. 为什么使用 CROSS JOIN

`CROSS JOIN` 用来生成笛卡尔积。

在本题中，它的作用是生成所有可能的：

```text
学生 - 科目
```

组合。

例如：

```text
3 个学生
3 门科目
```

会生成：

```text
3 × 3 = 9
```

行。

这样才能保证没有参加考试的学生-科目组合也会出现在最终结果中。

---

## 9. 为什么使用 LEFT JOIN

生成完整的学生和科目组合后，需要去 `Examinations` 表中匹配真实考试记录。

这里不能使用 `INNER JOIN`。

因为 `INNER JOIN` 只会保留匹配成功的记录。

如果某个学生没有参加某门考试，那么这组记录会被过滤掉。

本题要求保留参加次数为 0 的情况，所以必须使用：

```sql
LEFT JOIN
```

---

## 10. COUNT 的选择

本题建议使用：

```sql
COUNT(e.subject_name)
```

或者：

```sql
COUNT(e.student_id)
```

不建议使用：

```sql
COUNT(*)
```

原因是：

`LEFT JOIN` 后，即使没有匹配到考试记录，左表中的学生-科目组合仍然会保留一行。

如果使用：

```sql
COUNT(*)
```

那么没有参加考试的情况也会被统计为 1。

而使用：

```sql
COUNT(e.subject_name)
```

时，如果没有匹配到考试记录，`e.subject_name` 是 `NULL`。

`COUNT(字段名)` 不统计 `NULL`，所以结果会是 0。

## 5. 我的错误 SQL

```sql

FROM Employee

        FROM Employee
```

你的子查询是：

```sql

FROM Employee

HAVING COUNT(*) >= 5
```

它的结果可能返回多个 `managerId`。

例如：

```text
101
102
103
```

但外层使用的是：

```sql
WHERE id = (...)
```

`=` 后面的子查询应该只返回一个值。

如果子查询返回多个值，就会报错。

正确写法应该使用：

```sql
WHERE id IN (...)
```

因为我们要判断：

```text
员工 id 是否在这些 managerId 里面
```

---

错误点 2：没有排除 `managerId IS NULL`

你的子查询中：

```sql
GROUP BY managerId
```

会把 `NULL` 也分成一组。

如果有很多员工都没有经理，那么：

```text
managerId = NULL
```

这一组也可能满足：

```sql
HAVING COUNT(*) >= 5
```

虽然最终 `id IN (NULL)` 不会匹配出员工，但逻辑上最好明确排除：

```sql
WHERE managerId IS NOT NULL
```

这样子查询只统计真正的经理 ID。

---

错误点 3：`COUNT(*)` 本身不是最大问题，但要理解它的含义

在子查询中：

```sql
COUNT(*)
```

统计的是每个 `managerId` 出现了多少次。

也就是：

```text
有多少个员工的 managerId 等于这个经理 id
```

这正好表示这个经理有多少个直接下属。

所以在排除 `managerId IS NOT NULL` 后，`COUNT(*)` 可以使用。

也可以写成：

```sql
COUNT(managerId)
```

---

## 6. 正确 SQL：子查询写法

```sql
SELECT name
FROM Employee
WHERE id IN (
    SELECT managerId
    FROM Employee
    WHERE managerId IS NOT NULL
    GROUP BY managerId
    HAVING COUNT(*) >= 5
);
```

---

## 7. 解题思路：子查询写法

第一步，先统计每个经理有多少个直接下属。

```sql
SELECT managerId
FROM Employee
WHERE managerId IS NOT NULL
GROUP BY managerId
HAVING COUNT(*) >= 5
```

这里的逻辑是：

```text
按照 managerId 分组
每一组代表一个经理
COUNT(*) 统计这个经理有几个直接下属
HAVING COUNT(*) >= 5 筛选出至少有 5 个直接下属的经理
```

第二步，回到 `Employee` 表中找这些经理的姓名。

```sql
SELECT name
FROM Employee
WHERE id IN (...)
```

因为：

```text
Employee.id = 经理自己的员工 ID
Employee.managerId = 下属记录中的经理 ID
```

所以要用：

```sql
id IN (符合条件的 managerId)
```

---

## 8. 正确 SQL：自连接写法

```sql
SELECT
    e1.name
FROM Employee AS e1
JOIN Employee AS e2
ON e1.id = e2.managerId
GROUP BY
    e1.id,
    e1.name
HAVING COUNT(e2.id) >= 5;
```

---

## 9. 解题思路：自连接写法

把 `Employee` 表看成两张表：

```text
e1：经理表
e2：员工表
```

连接条件是：

```sql
e1.id = e2.managerId
```

含义是：

```text
e2 这个员工的 managerId 等于 e1 的 id
```

也就是说：

```text
e1 是 e2 的经理
```

连接后，一个经理有几个直接下属，就会出现几行。

然后按照经理分组：

```sql
GROUP BY e1.id, e1.name
```

再统计下属人数：

```sql
COUNT(e2.id)
```

最后筛选：

```sql
HAVING COUNT(e2.id) >= 5
```

## 10. 本题易错点总结

易错点 1：把 `id` 和 `managerId` 的关系看反

`id` 是员工自己的 ID。

`managerId` 是这个员工的经理 ID。

所以：

```text
某个人的 id 出现在别人的 managerId 中
```

才说明这个人是经理。

---

易错点 2：使用 `WHERE id = (子查询)`

如果子查询可能返回多个值，不能使用：

```sql
=
```

应该使用：

```sql
IN
```

---

易错点 3：忘记使用 `HAVING`

`WHERE` 是分组前过滤行。

`HAVING` 是分组后过滤组。

本题要筛选：

```text
直接下属人数 >= 5
```

这是聚合统计后的结果，所以必须使用：

```sql
HAVING COUNT(*) >= 5
```

---

易错点 4：没有排除 `managerId IS NULL`

`managerId` 为 `NULL` 表示没有经理。

它不是真正的经理 ID。

所以在子查询写法中，最好加上：

```sql
WHERE managerId IS NOT NULL
```
